# Local Model Agent Runtime State Spec

## Purpose

This spec turns the quick local-model agent research pass into an Atenas
architecture direction. The goal is not to adopt a large agent framework. The
goal is to borrow the useful architecture patterns that make weak local models
more reliable:

- durable checkpoints and resumable human-in-the-loop state
- model capability profiles
- context-budgeted prompt assembly
- deterministic toolset narrowing
- richer pending-action review
- trace replay for debugging and regression tests

This work should happen after, or alongside, the remaining v1 gaps in
`docs/superpowers/specs/2026-05-24-atenas-v1-gap-and-packaging-spec.md`.

## Research Signals

The useful patterns came from local and local-friendly agent ecosystems:

- LangGraph: durable execution, persistence/checkpoints, human-in-the-loop, and
  memory are first-class architecture concepts.
- OpenHands: local/open-weight models vary in tool reliability, so the runtime
  needs provider/model configuration, normalized errors, retries, and telemetry.
- LlamaIndex and CrewAI: agents work best when tools are explicit and schemas
  are structured; local models benefit from smaller, more relevant tool sets.
- LocalForge-style local agents: prompt assembly should respect a context
  budget and local model context window, especially when Ollama is the default.
- AutoGen and multi-agent systems: useful as prior art, but too much ceremony
  and latency for Atenas v1.

- OpenHands LLM architecture: https://docs.openhands.dev/sdk/arch/llm
- OpenHands LLM usage/local guidance: https://docs.openhands.dev/openhands/usage/llms

## Design Decision

Keep the current Atenas `AgentLoop` and `ToolRegistry`.

Do not migrate to LangGraph, CrewAI, AutoGen, or another orchestration runtime
for v1. Atenas has policy-sensitive local writes, Telegram-specific
confirmation, and a small enough loop that replacing it would add risk.

Instead, add small internal primitives that copy the useful ideas:

1. `AgentRuntimeStore` for durable thread state and pending actions.
2. `ModelProfile` for model-specific limits and JSON/tool reliability.
3. `ToolSelector` for deterministic toolset narrowing before prompt assembly.
4. `PromptAssembler` for context-budgeted prompts.
5. CLI and Telegram review helpers for pending actions and trace replay.

## Goals

- Pending confirm-first actions survive process restarts.
- A Telegram user can inspect, cancel, or confirm a pending action after a
  restart.
- The LLM sees only the tools relevant to the current message and state.
- Prompt history and observations are trimmed deterministically to fit a model
  profile.
- Local-model failures are clearer: invalid JSON, timeout, missing model, tool
  cap, and policy block are distinguishable in traces.
- Developers can replay a trace with mocked LLM responses or recorded tool
  observations to debug regressions.

## Non-Goals

- No multi-agent orchestration in v1.
- No remote dashboard/API write surface.
- No shell, arbitrary filesystem, source editing, or autonomous dependency
  installation tools.
- No full prompt/response logging by default.
- No replacement of `ToolRegistry` schemas with framework-native tool schemas.

## Proposed Components

### `core/nl/runtime_state.py`

Owns durable agent thread state:

- `AgentThreadState`
- `AgentCheckpoint`
- `PendingActionRecord`
- `AgentRuntimeStore`

Responsibilities:

- Store and load per-actor Telegram NL conversation state.
- Store pending confirm-first actions as JSON generated by Pydantic models.
- Mark pending actions as `pending`, `confirmed`, `cancelled`, `expired`, or
  `executed`.
- Never store full note/file body content or full prompts.
- Keep payloads redacted with the same payload summary rules used by action
  audit logs.

### SQLite Tables

Add tables owned by `core.db.SCHEMA_SQL`:

```sql
CREATE TABLE IF NOT EXISTS agent_threads (
    id TEXT PRIMARY KEY,
    actor_user_id INTEGER NOT NULL,
    channel TEXT NOT NULL DEFAULT 'telegram',
    status TEXT NOT NULL DEFAULT 'active',
    conversation_json TEXT NOT NULL DEFAULT '[]',
    selected_tools_json TEXT NOT NULL DEFAULT '[]',
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS pending_actions (
    id TEXT PRIMARY KEY,
    thread_id TEXT NOT NULL REFERENCES agent_threads(id),
    actor_user_id INTEGER NOT NULL,
    tool_name TEXT NOT NULL,
    action_type TEXT NOT NULL,
    proposal_json TEXT NOT NULL,
    confirmation_message TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'pending',
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    expires_at TEXT
);

CREATE INDEX IF NOT EXISTS idx_agent_threads_actor
ON agent_threads(actor_user_id, channel, status);

CREATE INDEX IF NOT EXISTS idx_pending_actions_actor_status
ON pending_actions(actor_user_id, status, created_at);
```

`pending_actions.proposal_json` stores the validated `ActionProposal` JSON, not
raw model output.

### `core/nl/model_profiles.py`

Defines local-model capabilities:

- model name pattern
- provider, initially `ollama`
- approximate context window
- recommended temperature
- request timeout
- supports strict JSON reliably
- supports native tool calling, currently false for the existing prompt path
- max tools to show per turn
- max history items
- max observation characters

The default profile should be conservative for Ollama:

```text
context_window_tokens: 8192
max_tools_per_prompt: 10
max_history_items: 8
max_observation_chars: 5000
strict_json: false
temperature: 0
```

### `core/nl/tool_selector.py`

Narrow the visible tools before prompt construction.

Inputs:

- user message
- recent conversation summary
- pending-action state
- full `ToolRegistry.list_tools()`

Outputs:

- ordered list of tool names to show to the LLM
- a selection reason for traces

Rules:

- If a pending action exists, show only review/cancel/confirmation context; do
  not ask the LLM to choose a new destructive action.
- Schedule/planning words select schedule, planning, assignments, classes,
  shifts, and availability tools.
- Notes/files/question words select search/retrieval/memory tools.
- Write verbs select the relevant read tools plus the matching act tools.
- Default fallback shows `get_status`, `get_today_overview`,
  `list_assignments`, `search_notes`, and `retrieve_sources`.
- Confirm-first destructive tools are included only when the message asks for a
  destructive or cleanup action.

This is a guardrail for weak local models, not a security boundary. Policy and
tool validation remain authoritative.

### `core/nl/prompt_assembler.py`

Builds the final agent prompt from:

- model profile
- selected tools
- cleaned conversation
- observations
- current user message

Responsibilities:

- Keep the existing `AGENT_PROMPT` semantics.
- Enforce maximum history and observation budgets.
- Produce prompt metadata for traces: selected tools, omitted tool count,
  history item count, and approximate character count.
- Do not include secrets or full source bodies beyond retrieval snippets.

### Telegram UX

Add deterministic slash commands:

- `/pending` shows the current pending action for the allowlisted user.
- `/cancel_pending` cancels the current pending action.

Plain `yes` / `no` remains supported. The pending action should be loaded from
SQLite first, with `context.user_data` kept only as a speed cache.

### CLI UX

Extend `atenas traces` later with:

- `atenas pending` to list pending actions.
- `atenas replay-trace TRACE_ID` to show trace steps and optionally run a dry
  replay with recorded observations.

## Data Flow

```text
Telegram text
  -> allowlist check
  -> load actor thread from AgentRuntimeStore
  -> if pending action exists:
       yes/no/pending/cancel are handled deterministically
  -> ToolSelector picks a small tool subset
  -> PromptAssembler builds budgeted prompt
  -> AgentLoop runs bounded tool calls
  -> after each tool call:
       trace step is written
       checkpoint is written
       pending actions are persisted
  -> final reply updates thread conversation
```

## Security Rules

- The LLM never writes runtime state directly.
- Runtime state is operational metadata, not a new user-data source of truth.
- Pending action execution still goes through `ToolRegistry.execute_pending()`
  and `ActionExecutor`.
- Confirmation belongs to the same actor who created the pending action.
- Unknown, expired, or actor-mismatched pending actions are cancelled or
  blocked, not executed.
- Full prompt/response text is not stored by default.

## Implementation Order

1. Add durable pending-action storage.
2. Wire Telegram confirmation to SQLite-backed pending actions.
3. Add `/pending` and `/cancel_pending`.
4. Add model profiles.
5. Add tool selection.
6. Add prompt assembly.
7. Add trace replay CLI.

The first three items are the highest-value path because they fix restart
safety for confirm-first actions.

## Acceptance Criteria

- A confirm-first action created by the agent is written to SQLite before the
  user sees the confirmation prompt.
- After rebuilding the Telegram context with empty `user_data`, replying `yes`
  executes the same pending action for the same actor.
- A different actor cannot confirm the pending action.
- `no` or `/cancel_pending` marks the pending action cancelled and does not
  mutate state.
- `/pending` shows the action type, affected record labels, and confirmation
  text without exposing sensitive payload content.
- Prompt assembly never shows more tools than the active model profile allows.
- Tool selection is deterministic for a seeded set of user messages.
- Existing agent-loop, policy, and Telegram NL tests keep passing.

## Recommended First Development Slice

Build only durable pending actions first. That slice gives Atenas the largest
local-model reliability gain with the smallest blast radius.

The slice includes:

- SQLite schema for `agent_threads` and `pending_actions`.
- `AgentRuntimeStore`.
- Telegram `natural_language_handler` loading/storing pending actions through
  the store.
- `/pending` and `/cancel_pending`.
- Tests for restart survival, actor mismatch, cancellation, and no mutation
  before confirmation.

Model profiles, tool selection, prompt assembly, and trace replay can follow as
separate slices.
